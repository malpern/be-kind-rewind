# Development Guide

This document covers the practical local workflow for working on Be Kind Rewind.

For quality expectations, also see:

- [quality-foundation-plan.md](./quality-foundation-plan.md)
- [agent-engineering-guide.md](./agent-engineering-guide.md)

## Project Shape

The package has three code targets and one current test target:

- `TaggingKit`: core library for storage, parsing, API clients, and classification plumbing
- `VideoTagger`: CLI for import, topic operations, metadata backfill, and OAuth flows
- `VideoOrganizer`: macOS app
- `TaggingKitTests`: current unit tests

In practice:

- core logic should generally land in `TaggingKit`
- CLI-specific wiring belongs in `VideoTagger`
- UI state and presentation belong in `VideoOrganizer`

## Requirements

- macOS 14+
- Swift 6.2 toolchain
- Xcode command line tools installed

Optional, depending on the task:

- Anthropic API key for topic suggestion and naming
- YouTube Data API key for metadata/channel enrichment
- Google OAuth desktop client config for OAuth-based YouTube flows

## Build

Build the package:

```bash
swift build
```

Build and package the app bundle:

```bash
./build-app.sh
```

The packaging script:

- builds `VideoOrganizerApp`
- copies the built binary into `Video Organizer.app`
- stops a running packaged app first if needed

Launch the packaged app:

```bash
open "Video Organizer.app"
```

## Run the CLI

Run the CLI through SwiftPM:

```bash
swift run video-tagger <subcommand> ...
```

Example:

```bash
swift run video-tagger suggest --inventory /path/to/inventory.json --topics 15
```

Playlist provenance refresh:

```bash
swift run video-tagger verify-all-playlists --db /tmp/full-tagger-v2.db
swift run video-tagger sync-pending --db /tmp/full-tagger-v2.db
swift run video-tagger browser-sync-login
swift run video-tagger import-seen-history --db /tmp/full-tagger-v2.db --file /path/to/watch-history.html
```

Use that command after importing or updating known playlists when you want the local database to know which playlists each saved video belongs to.

The CLI defaults to `video-tagger.db` in the current working directory unless a command overrides it with `--db`.

## Tests

Run the existing tests:

```bash
swift test
```

Local commits are guarded by a repo-managed pre-commit hook. When staged changes include `*.swift`, `Package.swift`, or `build-app.sh`, the hook runs `swift test` before the commit is created. Documentation-only commits skip the check automatically. If you need to bypass it for an intentional partial commit or local emergency, use `SKIP_LOCAL_CHECKS=1 git commit ...`.

Run tests with coverage:

```bash
swift test --enable-code-coverage
```

If you want a coverage summary for local inspection:

```bash
xcrun llvm-cov report \
  .build/arm64-apple-macosx/debug/VideoTaggerPackageTests.xctest/Contents/MacOS/VideoTaggerPackageTests \
  -instr-profile .build/arm64-apple-macosx/debug/codecov/default.profdata \
  Sources/TaggingKit/*.swift \
  Sources/VideoTagger/*.swift \
  Sources/VideoOrganizer/*.swift
```

Current reality:

- tests exist only for `TaggingKit`
- `VideoTagger` and `VideoOrganizer` still need dedicated test targets

OAuth client config is resolved from:

## Recommended Local Setup

Minimal setup for most work:

```bash
mkdir -p ~/.config/anthropic ~/.config/youtube
printf '%s\n' 'YOUR_ANTHROPIC_KEY' > ~/.config/anthropic/api-key
printf '%s\n' 'YOUR_YOUTUBE_API_KEY' > ~/.config/youtube/api-key
chmod 600 ~/.config/anthropic/api-key ~/.config/youtube/api-key
```

If working on OAuth flows, also place your Google desktop client JSON at:

```text
~/.config/youtube/oauth-client.json
```

If you need playlist provenance refreshes for private playlists, make sure OAuth is connected first so `video-tagger verify-all-playlists` can read them.
`sync-pending` uses the same OAuth connection for playlist-save actions. `Not Interested` stays in the queue as a browser-only deferred action until a browser executor is ready.
Browser-backed sync uses the dedicated Chrome profile at `~/.config/be-kind-rewind/playwright-profile`. Run `browser-sync-login` once, sign in to YouTube in that window, then later `sync-pending` or the app’s background sync loop can consume deferred browser actions. Failure screenshots/HTML are written to `output/playwright/browser-sync/`.

## Runtime Dependencies

- `./build-app.sh` is the supported packaging path; it signs the app bundle and bootstraps the managed discovery runtime
- discovery fallback installs `scrapetube` into `.runtime/discovery-venv` from `scripts/requirements-discovery.txt`
- the browser-sync Chrome profile is separate from your normal browser profile and is used for browser-only actions and API fallback

## Seen History Imports

The first-pass seen-history path is CLI-driven:

```bash
swift run video-tagger import-seen-history --db /tmp/full-tagger-v2.db --file /path/to/watch-history.html
```

Notes:

- supported input formats are best-effort `.json`, `.html`, `.htm`, and `.txt`
- the importer is designed for Google Takeout / My Activity exports
- exact `video_id` matches are stored in `seen_videos` and currently excluded from watch-candidate queries
- repeated imports are deduplicated conservatively by source plus `video_id`/URL and timestamp when available

## How We Work in This Repo

Default expectations:

- keep changes small and direct
- prefer modern Swift and SwiftUI patterns
- keep business logic out of view `body` code
- add tests for logic-heavy changes
- strengthen assertions when touching weak tests
- update docs when setup, workflow, or architecture changes

The detailed standards live in [agent-engineering-guide.md](./agent-engineering-guide.md).

## Development Priorities

When deciding where to put effort first:

1. protect `TaggingKit` logic with tests
2. make CLI behavior predictable and testable
3. extract and test pure logic from `VideoOrganizer`
4. avoid adding architecture that is heavier than the problem

## Before You Finish a Change

Use this lightweight checklist:

- does the code solve the real problem?
- is the change simpler or clearer than before?
- did you run the relevant tests?
- if behavior changed, are the assertions specific?
- if setup or workflow changed, did you update docs?
- did you avoid unnecessary abstraction or infra?
